# MemoryOps memory-governance benchmark

A public, reproducible benchmark that **measures** memory governance instead of
claiming it. Most memory systems assert "we delete", "we isolate tenants", "we don't
leak". This scores those claims on named suites against an isolated, offline stack —
so you can reproduce the numbers and score your own memory system on the same axes.

## Run it

```bash
python benchmark/run_benchmark.py           # human scorecard + leaderboard
python benchmark/run_benchmark.py --json     # machine-readable scorecard
python benchmark/run_benchmark.py --md benchmark/SCORECARD.md
```

No API keys, no infra — the cases run against the deterministic stub stack (`evals/`).

### Run it in Docker

If you'd rather not set up Python locally, the repo ships an opt-in Compose profile
that runs the same scorecard in an isolated container (offline, in-memory, no DB/Redis):

```bash
docker compose --profile benchmark run --rm benchmark
```

The `benchmark` profile keeps it out of the default `docker compose up` stack — it's a
one-shot run that exits non-zero if a critical suite isn't perfect.

## Suites

| Suite | What it proves | Critical |
| --- | --- | --- |
| **deletion_and_leakage** | deleted/expired memory is not retrieved and cannot leak — directly, indirectly, cross-session, via summaries, or after reindex/rebuild | ★ |
| **tenant_isolation** | no cross-tenant / cross-user retrieval | ★ |
| **context_admission** | only relevant *and* allowed memory enters context; archived/temporary paths behave | |
| **policy_governance** | policy-before-storage: secrets/injection BLOCK, trivia DROP, sensitive PENDING, real preferences SAVE | |
| **retrieval_quality** | hybrid retrieval with an explainable score breakdown | |

**Critical suites must be perfect.** The benchmark exits non-zero if
`deletion_and_leakage` or `tenant_isolation` is not 100%, or if the overall pass rate
is below 80%.

## Current scorecard

See [SCORECARD.md](SCORECARD.md) (regenerated by the runner). The deletion-leakage
suite is the headline: MemoryOps is one of the few memory systems that *tests* whether
deleted memory can still influence output (v1.4/v1.5), not just whether the row is gone.

## Extraction quality (honest instrument)

The governance suites above run against the deterministic **stub** so they are
reproducible and offline. Extraction *quality* is scored separately, per provider, in
[EXTRACTION_QUALITY.md](EXTRACTION_QUALITY.md) — precision/recall over
[hand-labeled turns](../evals/datasets/extraction_golden.jsonl):

```bash
python evals/run_extraction_quality.py                     # stub only (offline)
python evals/run_extraction_quality.py --provider openai   # needs OPENAI_API_KEY
```

The stub is documented as a **test fixture, not the product** — it is precise but
trails real models on recall and multi-memory turns, and showing that gap is the
point. Real-provider behavior is proven deterministically in CI via committed VCR
cassettes (`services/api/tests/test_provider_recorded.py`), and the
`nightly-live.yml` workflow re-checks a real provider daily when an API-key secret is
configured (it skips cleanly otherwise).

## Bring your own system

The suites are defined by outcome, not implementation. To benchmark another memory
system, implement the same case kinds (`evals/*.json` documents them) against your
system and report the per-suite pass rates — the "deletion leakage leaderboard" is
simply this table filled in for each entrant.
